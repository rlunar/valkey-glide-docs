## What is covered in this tutorial

In our first PubSub tutorial, you built a listener that used an `await` loop to "pull" messages from the client. This is a great, simple model, but Valkey GLIDE also offers a more powerful "push" model using **callbacks**.

[cite\_start]A callback is a Python function you write and pass to the GLIDE client during configuration[cite: 27, 41]. [cite\_start]When a new message arrives on a subscribed channel, the GLIDE client will *automatically* execute your function, passing the message data to it[cite: 25, 28]. This approach is often cleaner, as you don't need to manage an async loop yourself.

By the end of this tutorial, you will refactor the `listener.py` script from the "Hello World" example to use a callback. You will also learn how to pass user-defined "context" to your callback, a powerful feature for managing application state.

## Steps to Refactor Your Listener for Callbacks

Our original `listener.py` script had a `while True:` loop that manually called `await client.get_pubsub_message()`. We will now replace that entire block with a callback function.

### Step 1: Define Your Callback Function

First, we need to create the function that GLIDE will call. This function must accept two arguments:

1.  `msg`: A `PubSubMsg` object containing the message, channel, and pattern.
2.  [cite\_start]`context`: Any user-defined data you passed in during configuration[cite: 40]. This is perfect for passing application state, database connections, or, in our case, a "shutdown" signal.

Let's create a new file named `listener_callback.py`.

At the top of the file, add your imports and define a global `asyncio.Event`. We'll use this event to tell our main function when it's time to shut down.

```python
import asyncio
from typing import Any
from valkey_glide import (
    GlideClient,
    GlideClientConfiguration,
    NodeAddress,
    PubSubChannelModes,
    PubSubSubscriptions,
    CoreCommands,
)

# This event will help us gracefully shut down the listener
shutdown_event = asyncio.Event()

def on_message_callback(msg: CoreCommands.PubSubMsg, context: Any):
    """
    This is the function GLIDE will call for every message.
    
    IMPORTANT: This callback is executed in a background thread
    managed by the client, not on the main asyncio event loop.
    """
    
    # The 'context' is whatever we passed in the client config
    print(f"Context: {context}")
    print(f"Received Message: '{msg.message}' on Channel: '{msg.channel}'")
    print("---")
    
    # If we get an 'exit' message, set the event
    if msg.message == "exit":
        print("Exit message received. Setting shutdown event.")
        # This is thread-safe and will signal our main() function
        shutdown_event.set()
```

> **Important Note on Threading**
>
> [cite\_start]The documentation states that your callback code must be thread-safe[cite: 26]. The GLIDE client executes this function in a separate, internal thread, *not* on the main Python `asyncio` event loop.
>
> Using `asyncio.Event.set()` is thread-safe and is the correct way to communicate from the callback thread back to your main `asyncio` task.

### Step 2: Configure the Client to Use the Callback

Now, let's write the `main` function for `listener_callback.py`. The setup is similar, but the `PubSubSubscriptions` configuration is different.

We will pass our `on_message_callback` function and a simple string `"my-listener-v1.0"` as our context.

```python
async def main():
    """
    Creates a dedicated PubSub client that uses a callback
    and waits for a shutdown signal.
    """
    print("Starting callback listener client...")
    
    # 1. Define the context we want to pass to our callback
    app_context = "my-listener-v1.0"
    
    # 2. Configure subscriptions, this time with our callback
    subscriptions = PubSubSubscriptions(
        channels_and_patterns={
            PubSubChannelModes.Exact: {"channel:1"},
            PubSubChannelModes.Pattern: {"news:*"}
        },
        # Pass the function *reference*
        callback=on_message_callback,
        # Pass our application-specific context
        context=app_context,
    )

    # 3. Create the client configuration
    config = GlideClientConfiguration(
        addresses=[NodeAddress("localhost", 6379)],
        pubsub_subscriptions=subscriptions,
    )

    # 4. Create the client
    client = await GlideClient.create(config)
    print("Listener connected. Callbacks will now fire on new messages.")
    print("Waiting for 'exit' message...")

    try:
        # 5. Wait for the shutdown event
        # The client is listening in the background.
        # Our main coroutine just needs to wait until
        # the callback tells it to exit.
        await shutdown_event.wait()
        
    except Exception as e:
        print(f"An error occurred: {e}")
    finally:
        # 6. Close the client to unsubscribe
        print("Closing listener client.")
        await client.close()

if __name__ == "__main__":
    # Add the callback function and main() to the
    # *same file* (listener_callback.py)
    # and run it.
    asyncio.run(main())
```

> **Warning: Do Not Mix Receiving Methods**
>
> When you configure a client with a callback, you **cannot** use the polling or async methods (`try_get_pubsub_message` or `get_pubsub_message`). [cite\_start]Any calls to those methods will fail[cite: 29]. The client is now in "push" mode, and all messages are sent *only* to your callback.

### Step 3: Run Your Application

This is the fun part. The process is the same as before, but the result is cleaner.

**In Terminal 1:** Start your new callback listener.

```bash
python listener_callback.py
```

You will see it connect and wait:

```
Starting callback listener client...
Listener connected. Callbacks will now fire on new messages.
Waiting for 'exit' message...
```

**In Terminal 2:** Run the *exact same* `publisher.py` script from the "Hello World" tutorial.

```bash
python publisher.py
```

You will see it publish its messages as normal:

```

Publisher connected.
Published 'Hello World!' to 'channel:1'
Published 'GLIDE for Python is here!' to 'news:python'
Published 'exit' to 'channel:1'
Closing publisher client.
```

**Look back at Terminal 1\!** You will see your callback function fired for each message, printing the context you provided, and then shutting down gracefully when it received "exit".

## Summary and Next Steps

Congratulations\! You have successfully refactored your listener to a fully event-driven, "push" model.

  * [cite\_start]You learned how to **define a callback function** that accepts a message and a context[cite: 40].
  * [cite\_start]You **configured `PubSubSubscriptions`** to use your callback[cite: 41].
  * You saw how the client **runs the callback in a background thread**, allowing your main `asyncio` task to focus on other work (or, in this case, just wait for a signal).
  * You used an `asyncio.Event` for **thread-safe communication** from your callback back to your main coroutine.

This pattern is far more efficient than polling and is the recommended way to handle real-time PubSub data.
